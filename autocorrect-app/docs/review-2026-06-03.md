# autocorrect-app 项目审查报告

> 审查日期: 2026-06-03
> 审查方向: 类 Grammarly 语法校验 / AI 能力 / 系统原生无障碍输入

---

## 1. 项目架构速览

| 层级      | 技术栈                                                     | 职责                               |
| --------- | ---------------------------------------------------------- | ---------------------------------- |
| 前端      | Tauri 2 + Svelte 5 + TS + Tailwind 4                       | 设置面板、拼写检查器、AI 工具 UI   |
| Rust 后端 | Tauri commands + 原生线程                                  | IPC、无障碍 API 调用、热键、覆盖层 |
| 核心库    | `autocorrect` crate + `typos` crate                        | CJK 格式化规则、字典拼写检查       |
| 系统层    | macOS Accessibility API (`AXUIElement`) + `rdev` + `enigo` | 全局文本获取、热键监听、输入模拟   |

**关键文件映射**

- `macos_text.rs` — AXUIElement 封装（文本获取、坐标计算、选区操作）
- `overlay.rs` — 原生 NSView 下划线覆盖层
- `popup.rs` / `ai_popup.rs` — 拼写建议弹窗 / AI 图标弹窗
- `ai_grammar.rs` — 全部 AI 命令（grammar/translate/polish/tone/clarity/vocabulary/assist）
- `hotkey.rs` — rdev 全局热键
- `lib.rs` — 主事件循环（800ms typo 同步轮询 + 100ms hover 检测）

## 2. 方向一：类 Grammarly 语法校验

### 现有能力

- **CJK 规则引擎**：`autocorrect::format_for` + `lint_for` 处理空格、全半角标点、引号等
- **字典拼写检查**：`typos` crate 本地检测错字，返回 `TypoSuggestion`（含建议词列表）
- **行级 diff**：`LineChange` 追踪每处变更的行列位置
- **视觉覆盖层**：原生 NSView 渲染 wavy/solid/dashed/dotted 下划线，颜色可配置
- **交互闭环**：hover 下划线 → 弹出建议窗 → 点击替换 / 加入自定义词库
- **应用级开关**：`ignored_apps` 黑名单，避免在特定 App 中触发

### 差距与问题

| 问题                  | 严重程度 | 说明                                                                              |
| --------------------- | -------- | --------------------------------------------------------------------------------- |
| 无真正的语法检查      | **高**   | 现有能力停留在"拼写+CJK 格式"，无**主谓一致、时态、冠词、从句结构**等语法分析     |
| AI grammar 是整段重写 | **高**   | `ai_grammar_check` 返回全新文本，**不返回带位置标记的问题列表**，用户无法逐条审阅 |
| 无实时边写边检        | **中**   | 依赖 800ms 轮询，非事件驱动；Grammarly 是字符级实时                               |
| 无上下文感知          | **中**   | `typos` 基于字典，无法区分 `there/their/they're` 等同音异义                       |
| 无文档级分析          | **中**   | 无 readability score、无写作目标、无风格一致性检查                                |
| 无个人词库学习        | **低**   | 自定义词库是静态文件，不会根据用户接受/拒绝建议自动学习                           |

### 建议

1. **引入结构化语法分析层**：对接 `languagetool` 或自研轻量规则引擎，生成带 `line/col` 的语法问题列表
2. **AI grammar 返回标记而非重写**：让 LLM 输出 `{"issues":[{"type":"grammar","message":"...","suggestion":"...","offset":N}]}`, 与本地 typo 统一渲染
3. **接入 Accessibility Notifications**：监听 `AXSelectedTextChanged`、`AXValueChanged`, 减少轮询到事件驱动
4. **增加 readability 指标**：集成本地可读性算法（Flesch-Kincaid、SMOG），不依赖 API

## 3. 方向二：AI 能力

### 现有能力

- **7 个 AI 命令**：grammar / translate / polish / tone-detect / clarity-check / vocabulary-enhance / assist
- **2 种传输模式**：同步 (`ai_text_transform`) + SSE 流式 (`ai_text_transform_stream`), 流式有完整 guardrail
- **多风格并行润色**：`ai_polish_batch` 用 `tokio::spawn` 并行请求 formal/conversational/academic/business
- **Grammarly 风格 assist**：`ai_assist` 返回 headline + summary + primaryText + alternatives + focus tags
- **OpenRouter 兼容**：可配置 endpoint / model / API key，默认 `gpt-5-nano` via OpenRouter
- **原生 AI 图标**：选中文本后屏幕出现 💡 NSPanel，hover 展开 AI 操作面板
- **流式日志详尽**：request_id 追踪、chunk 计数、字节统计、reasoning-only 检测、stall guard

### 差距与问题

| 问题                          | 严重程度 | 说明                                                          |
| ----------------------------- | -------- | ------------------------------------------------------------- |
| 全部依赖远程 API              | **高**   | 无本地模型；断网/高延迟时 AI 功能完全不可用                   |
| 无请求缓存/去重               | **高**   | 同样文本、同样操作会重复发请求；快速输入可能并发多请求        |
| clarity/vocabulary 前端未接入 | **中**   | 后端命令存在，但 `SpellChecker.svelte` 无调用入口，功能"沉睡" |
| 无用户反馈回路                | **中**   | 无 thumbs up/down，无法收集偏好用于后续 prompt 调优           |
| 无 API 用量管理               | **中**   | 无 token 计数、无速率限制提示、无月度用量统计                 |
| AI 与本地 typo 视觉层级混淆   | **低**   | 都是下划线，无图标/颜色区分"AI 建议" vs "本地拼写错误"        |
| 无 Smart Compose              | **低**   | 无 predictive text / 自动补全建议                             |

### 建议

1. **增加本地轻量模型兜底**：集成 `llama.cpp` 或 `candle` 加载 1B-3B 模型，处理常见 grammar/clarity（断网可用）
2. **请求缓存层**：LRU 缓存 `(text_hash, operation) → result`，TTL 5 分钟；防抖 300ms 避免输入抖动
3. **前端唤醒 clarity/vocabulary**：在 SpellChecker 增加"分析清晰度"、"增强词汇"按钮，调用已有命令
4. **用户反馈收集**：AI 建议卡片增加 👍/👎，写入本地 SQLite，定期批量用于 prompt A/B
5. **API 用量追踪**：解析响应 header / 估算 token，展示在设置面板（本月已用 / 预估费用）

## 4. 方向三：系统原生无障碍输入

### 现有能力

- **完整的 AXUIElement 封装**：`AXFocusedUIElement` → `AXValue` / `AXSelectedText` / `AXSelectedTextRange` / `AXBoundsForRange`
- **AXPollSession 缓存**：每轮轮询只获取一次焦点元素，复用于文本、坐标、选区查询
- **Electron/CEF 兼容**：`is_window_relative_coords` + `window_pos` 补偿，修正 Slack/VS Code 等坐标偏差
- **大文本分片**：>20000 UTF-16 时截断到 caret 前后 4000 单位，**词边界对齐**（`snap_utf16_to_word_start/end`）
- **剪贴板回退**：AX API 失败时用 `osascript` 模拟 Cmd+C 获取选中文本
- **输入模拟**：`enigo` 模拟 Cmd+V 粘贴修正结果
- **原生覆盖层**：NSView 下划线 + NSPanel AI 图标，非 WebView，性能与 z-index 更优
- **权限管理**：`AXIsProcessTrusted` 检测 + 引导跳转 System Settings

### 差距与问题

| 问题                       | 严重程度 | 说明                                                                                                     |
| -------------------------- | -------- | -------------------------------------------------------------------------------------------------------- |
| **仅 macOS**               | **高**   | Windows 无 UIA/MSAA、Linux 无 AT-SPI，产品天花板受限                                                     |
| clipboard monitor 是占位   | **高**   | `start_clipboard_monitor` / `stop_clipboard_monitor` 只有 `emit`, 无实际后台线程                         |
| 无事件驱动，纯轮询         | **中**   | 800ms 轮询 + 100ms hover；无法监听 `AXValueChanged`, CPU/延迟权衡差                                      |
| 无 Apple Intelligence 集成 | **中**   | macOS 15+ 有 Writing Tools API，可作为更高阶体验的备选路径                                               |
| 无输入法模式               | **中**   | 当前是"外挂"模式（热键触发），无法像输入法那样随键即时响应                                               |
| 无 undo/redo               | **中**   | 替换文本后无撤销栈，用户误替换无法一键恢复                                                               |
| 无光标位置恢复             | **低**   | 粘贴后光标跳到文末，未恢复原光标位置                                                                     |
| 无 sandbox 支持            | **低**   | 若上 Mac App Store，需要 `com.apple.security.temporary-exception.mach-lookup.global-name` 等 entitlement |

### 建议

1. **实现真正的 clipboard monitor**：复用 `create_clipboard_channel` 已有实现，替换 placeholder 命令
2. **探索 Apple Intelligence Writing Tools**（macOS 15+）：`WKWebView` / `UITextInput` 的 `writingTools` 属性可作为苹果官方能力的 fallback
3. **增加 Keyboard Event Tap**：在 CGEventTap 层拦截按键，实现"输入即检"而非热键触发；需处理权限与性能
4. **光标位置恢复**：粘贴后通过 `AXSelectedTextRange` 计算原位置偏移量，恢复光标
5. **增加 undo 支持**：替换前将原文写入内部栈，热键或 UI 提供撤销上一步
6. **跨平台抽象层**：定义 `PlatformAccessibility` trait，macOS 用 AXUIElement，Windows 用 UI Automation，Linux 用 AT-SPI

## 5. 综合风险评估

| 风险                      | 等级   | 影响                                                       |
| ------------------------- | ------ | ---------------------------------------------------------- |
| macOS-only 限制产品市场   | **高** | 无法覆盖 Windows 办公场景，商务用户流失                    |
| 纯云端 AI 导致体验不稳定  | **高** | 网络波动时核心卖点失效，用户留存差                         |
| 轮询架构性能与响应瓶颈    | **中** | 800ms 延迟让"实时感"缺失，对比 Grammarly 处于劣势          |
| AX API 被目标 App 限制    | **中** | 越来越多的 App（如银行、密码管理器）拒绝 AX 访问，功能失效 |
| 无本地模型 = 数据隐私顾虑 | **中** | 企业用户敏感文本不愿上云，需本地能力背书                   |

## 6. 优先级路线图建议

### P0（立即）

- AI grammar 从"整段重写"改为"结构化 issue 列表"，与本地 typo 统一标记渲染
- 接入 `ai_clarity_check` / `ai_vocabulary_enhance` 到前端 UI，唤醒沉睡功能
- 实现真正的 clipboard monitor（已有 `clipboard.rs`，只需替换 placeholder）

### P1（短期）

- 增加请求缓存 + 防抖，减少重复 AI 调用
- 引入本地轻量 LLM（1B-3B）作为离线兜底
- 光标位置恢复 + undo 支持

### P2（中期）

- 探索 Keyboard Event Tap 实现输入级实时检测
- 接入 Apple Intelligence Writing Tools（macOS 15+）
- Windows UIA 调研与 POC

### P3（长期）

- 用户反馈收集系统（RLHF 轻量版）
- 个人写作风格学习模型
- Smart Compose / predictive text
